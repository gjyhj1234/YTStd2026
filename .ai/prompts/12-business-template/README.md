# 业务模块模板提示词

## 概述

本目录包含**可复用的业务模块开发模板提示词**。当需要开发新的业务系统（如 SaaS 平台、运营后台、客户端应用等）时，复制本目录下的模板到对应的业务提示词目录下，填写业务特定内容即可。

---

## 设计理念

### 可复用层（基础设施）

以下内容在所有业务中共享，**不需要为每个业务重写**：

| 层 | 提示词位置 | 说明 |
|----|-----------|------|
| 底层框架 | `.ai/prompts/11-base-library/` | YTStdSqlBuilder、YTStdLogger、YTStdAdo、YTStdEntity、YTStdI18n 及其 Generator |
| 后端通用 | `.ai/prompts/02-backend/` | 实体建模、应用服务、端点、中间件的通用规范 |
| 前端通用 | `.ai/prompts/03-frontend/` | 前端治理、任务拆分、DevExtreme 规范、axios 规范、i18n 执行规范、业务模板 |
| 数据库通用 | `.ai/prompts/04-database/` | 表结构设计、种子数据的通用规范 |
| 国际化通用 | `.ai/prompts/05-i18n/` | 后端/前端国际化的通用规范 |
| 生成器通用 | `.ai/prompts/06-generator/` | 实体生成器、CRUD 生成器的通用规范 |
| 测试通用 | `.ai/prompts/07-testing/` | 单元测试、集成测试、Postman 的通用规范 |

### 业务专用层

以下内容是每个业务系统**独立拥有**的，需要在业务目录下单独定义：

| 内容 | 说明 |
|------|------|
| 数据库设计 | 业务表结构、索引、字段 |
| 实体定义 | 业务实体、枚举、值对象 |
| API 设计 | 业务端点、DTO、错误码 |
| 后端服务 | 业务逻辑、状态机、校验 |
| 前端页面 | 业务页面、组件、路由 |
| 初始化数据 | 业务种子数据 |
| 测试 | 业务测试用例 |

## 目录结构约定

每个业务系统在 `prompts/` 下占用一个编号目录，内部结构如下：

```
.ai/prompts/{NN}-{business}/
├── README.md                    # 业务总览与任务索引
├── database/                    # 数据库设计提示词
│   ├── schema.md                # 表结构设计
│   └── seed-data.md             # 初始化数据
├── backend/                     # 后端提示词
│   ├── infrastructure.md        # 后端基础设施（Program.cs、中间件、认证）
│   ├── {module}-api.md          # 各业务模块 API（每个模块一个文件）
│   └── error-codes.md           # 错误码定义
├── frontend/                    # 前端提示词（00xx_xxx.md 命名规范）
│   ├── 0000_overview.md         # 前端总览
│   ├── 0001_scaffold.md         # 前端工程脚手架
│   ├── 0002_i18n.md             # 前端国际化接入
│   ├── 0010_layout.md           # 布局与导航
│   ├── 0011_login-page.md       # 登录页
│   └── 00xx_{module}-page.md    # 各模块页面（每个模块一个文件）
└── testing/                     # 测试提示词
    ├── backend-tests.md         # 后端测试
    └── postman.md               # Postman 集合
```

## 新业务接入流程

### 第 1 步：创建业务目录

```bash
cp -r .ai/prompts/12-business-template/template/ .ai/prompts/{NN}-{business}/
```

### 第 2 步：填写 README.md

- 业务名称与目标
- 技术栈（复用底层框架 + 业务前后端工程）
- 后端项目路径（如 `src/YTStd{Business}/`）
- 前端项目路径（如 `web/{business}-web/`）
- 模块列表与执行顺序

### 第 3 步：编写数据库设计

- 在 `database/schema.md` 中定义业务表结构
- 确认遵循 `.ai/rules/database.md` 命名规范

### 第 4 步：编写后端提示词

- 每个业务模块一个 `backend/{module}-api.md`
- 在 `backend/error-codes.md` 中定义业务错误码

### 第 5 步：编写前端提示词

- 每个前端模块一个 `frontend/{module}-page.md`
- 在 `frontend/i18n.md` 中定义国际化接入计划

### 第 6 步：编写测试提示词

- 定义测试策略和覆盖目标

## 关键原则

1. **通用规范在通用目录**：不要在业务提示词中重复通用规范，只引用
2. **业务细节在业务目录**：业务表结构、API、页面等细节只写在业务目录下
3. **每模块一文件**：便于 Agent 单次执行，减少上下文负担
4. **前后端独立文件**：前端和后端提示词分开，可以分别交给不同 Agent 执行
5. **统一引用通用规则**：每个业务提示词的"前置阅读"中引用对应的通用规则
