# 前端提示词迭代与缺陷沉淀 workflow

> 本文件定义“修一次缺陷，如何让下一次不再重犯”。如果只修代码、不修提示词体系，等于没有真正解决问题。

---

## 一、触发条件

出现以下任一情况，必须触发提示词迭代：

1. 需求漏项导致功能缺失。
2. 同类组件被重复实现。
3. DevExtreme 能力使用不完整。
4. 权限入口、按钮或路由逻辑失真。
5. 测试宣称通过，但缺少证据或遗漏关键场景。
6. 同类问题第二次出现。

---

## 二、缺陷分类与回写目标

| 缺陷类型 | 根因 | 必须回写文件 |
| --------- | ------ | ------------- |
| 需求漏项 | 提示词没有默认补全 | `09-production-defaults.md` |
| 复用不足 | 缺少资产目录或资产升级规则 | `10-component-asset-catalog.md` |
| 组件能力遗漏 | DevExtreme 能力矩阵不完整 | `04-devextreme-templates.md` |
| 业务模板过粗 | 模块提示词或模板不够细 | `07-business-prompt-template.md` 或具体业务提示词 |
| 测试伪通过 | E2E 规范缺少证据约束 | `08-playwright-e2e.md` 或 `0040_e2e-testing-protocol.md` |
| 续接丢失 | 任务快照和缓存未强制 | `11-delivery-workflow.md` |
| GitHub 队列失灵 | 切片、标签或顺序规则不合理 | `12-github-automation-workflow.md` |
| 通用坏味道 | 需要长期提醒 | `03-anti-patterns.md` |

---

## 三、每次修复后的最小回写动作

每次修复一个缺陷，至少完成以下 3 件事：

1. 修代码或修提示词内容本身。
2. 在 session summary 中记录“问题 -> 根因 -> 回写文件”。
3. 更新至少一处提示词或 workflow 文件。

如果没有第 3 步，视为沉淀不完整。

---

## 四、回写判断清单

- [ ] 这个问题是否可能再次出现？
- [ ] 这个问题是否能通过默认能力预防？
- [ ] 这个问题是否说明某个共享资产应被抽出来？
- [ ] 这个问题是否说明测试矩阵缺项？
- [ ] 这个问题是否说明切片太大或 GitHub issue 编排不合理？

只要任一项为“是”，就必须更新提示词体系。

