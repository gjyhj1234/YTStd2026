# 业务实施提示词模板（生产级、可验收版）

> 本模板不是“写个页面”模板，而是“写一个可以被 code review、E2E、续接、GitHub issue 队列共同消费”的实施合同。

---

## 一、强制章节

每个业务模块提示词必须包含以下章节，禁止省略：

```markdown
# {业务系统} — {模块名称}

## 任务信息
## 当前执行模型（Epic / Slice）
## 前置阅读
## 需求补全矩阵
## 缺口与待确认
## 组件复用决策表
## DevExpress 文档查阅
## DevExtreme 能力矩阵
## API 端点
## 必须产出的文件
## 页面结构
## 可视化设计审查包
## 查询功能
## 列表与分页
## 权限入口可见性矩阵
## 操作按钮
## 表单功能
## 静态配置文件
## 国际化要求
## E2E 测试证据矩阵
## 任务快照与缓存更新要求
## 验收标准
```

## 二、关键章节写法要求

### 2.1 任务信息

```markdown
## 任务信息

| 属性 | 值 |
|------|---|
| Epic 编号 | F2-2 |
| 当前 slice | F2-2C |
| 所属阶段 | 层级 2：业务页面层 |
| 依赖 slice | F2-2A、F2-2B |
| 预计文件数 | 4-6 个 |
| 预计耗时 | 25-35 分钟 |
```

### 2.2 需求补全矩阵

必须明确以下 4 列，不能只写“按默认实现”：

```markdown
## 需求补全矩阵

| 类别 | 显式需求 | 生产默认补全 | 待确认项 | 明确不做项 |
|------|---------|-------------|---------|-----------|
| DataGrid 能力 | 远程分页 | 焦点行、移动端列隐藏 | 是否启用列选择器 | 导出 |
| 表单 | 新增/编辑弹窗 | 提交 loading、失败不关闭 | 是否需要密码确认 | 批量导入 |
```

### 2.3 组件复用决策表

```markdown
## 组件复用决策表

| 功能点 | 复用资产 | 当前仓库状态 | 决策 | 理由 |
|--------|---------|-------------|------|------|
| 页面头部 | FunctionDescriptionCard / OperationGuideDrawer | 已存在 | 复用 | 页面级统一 |
| 搜索区 | SearchPanel | 不足 | 新建共享资产 | 第二个模块开始重复 |
```

### 2.4 可视化设计审查包

这是新增的强制章节，目标不是替代正文，而是让 AI 和人工在编码前先看见页面骨架、关键交互、字段覆盖和状态反馈，快速发现遗漏项与不合理设计。

必须同时包含以下 5 类产物，禁止只给纯文本描述：

```markdown
## 可视化设计审查包

### 2.4.1 页面结构蓝图

~~~mermaid
flowchart TB
  Page["平台用户管理页"]
  Page --> Header["标题/副标题<br/>功能说明卡片<br/>操作指引抽屉"]
  Page --> Query["查询区<br/>关键词 + 状态"]
  Page --> Advanced["高级查询<br/>角色 + 时间范围"]
  Page --> Toolbar["工具栏<br/>新增 + 批量操作"]
  Page --> Grid["列表区<br/>DataGrid + Pager"]
  Page --> CreatePopup["新增弹窗"]
  Page --> EditPopup["编辑弹窗"]
  Page --> DetailPopup["详情抽屉/弹窗"]
~~~

### 2.4.2 关键交互流

~~~mermaid
flowchart LR
  Enter[进入页面] --> Perm{"具备 list 权限?"}
  Perm -- 否 --> Forbidden[403 / 无权限提示]
  Perm -- 是 --> Load[加载首屏数据]
  Load --> Search[查询 / 重置]
  Search --> Grid[刷新列表]
  Grid --> Create[新增]
  Grid --> Edit[编辑]
  Grid --> Detail[查看详情]
  Grid --> Danger[启用 / 禁用 / 删除]
  Create --> ValidateCreate{"表单校验通过?"}
  ValidateCreate -- 否 --> CreateStay[保留弹窗并显示校验]
  ValidateCreate -- 是 --> SubmitCreate[提交创建 API]
  SubmitCreate --> RefreshCreate[关闭弹窗 刷新列表 定位记录]
~~~

### 2.4.3 字段覆盖矩阵

| 字段 | 查询 | 列表 | 详情 | 新增 | 编辑 | 必填 | 只读 | 校验/说明 |
| ------ | ------ | ------ | ------ | ------ | ------ | ------ | ------ | ----------- |
| Username | - | ✅ | ✅ | ✅ | ✅ | 新增必填 | 编辑只读 | 唯一性 / 长度 / pattern |
| RoleIds / Roles | 高级查询可选 | ✅ | ✅ | ✅ | ✅ | 新增/编辑必填 | 否 | 至少一个角色 |

### 2.4.4 操作/API 对照矩阵

| 操作 | 入口 | 权限 | API | 确认 | 成功反馈 | E2E |
| ------ | ------ | ------ | ----- | ------ | ---------- | --- |
| 新增 | 工具栏 | `{module}.create` | `POST /api/...` | 否 | `notifySuccess('创建成功')` | ✅ |
| 编辑 | 行操作 | `{module}.update` | `PUT /api/.../{id}` | 否 | `notifySuccess('更新成功')` | ✅ |
| 删除 | 行操作 | `{module}.delete` | `DELETE /api/.../{id}` | 是 | `notifySuccess('删除成功')` | ✅ |

### 2.4.5 状态反馈矩阵

| 场景 | 触发条件 | UI 承载 | 用户可见反馈 | 恢复方式 |
| ------ | --------- | --------- | ------------- | --------- |
| 首屏加载 | 首次进入页面 | LoadPanel / Skeleton | 加载中 | 自动恢复 |
| 空数据 | 查询结果为空 | DataGrid empty text | 暂无数据 | 调整查询条件 |
| 无权限 | 无 list 权限 | 403 页面 | 无权限提示 | 返回上级或切换账号 |
| 提交失败 | API 失败或校验失败 | 弹窗内表单 | 错误提示且不关闭 | 修改后重试 |
```

补充要求：

1. 页面上出现的区域、按钮、弹窗、抽屉，必须都能在蓝图或交互流中找到。
2. 表单字段、列表列、详情字段，必须都能在字段覆盖矩阵中找到。
3. 所有会调用 API 的交互，必须都能在操作/API 对照矩阵中找到。
4. 加载、空态、错误、403、提交中等关键状态，必须都能在状态反馈矩阵中找到。
5. 业务提示词生成时，必须先完成这套审查包，再继续写实现章节和验收标准。

### 2.5 DevExtreme 能力矩阵

只要使用到组件，就必须有矩阵：

```markdown
## DevExtreme 能力矩阵

### DxDataGrid

| 能力 | 状态 | 配置 |
|------|------|------|
| 远程分页 | 启用 | `CustomStore` |
| 默认排序 | 启用 | `CreatedAt desc` |
| 固定列 | 启用 | `操作列 fixed right` |
| 列隐藏 | 启用 | 移动端 `hidingPriority` |
| 列显示/隐藏 | 禁用 | 本页不开放 chooser |
| 焦点行 | 启用 | 保存后定位 |
```

### 2.6 权限入口可见性矩阵

这是必须新增的章节，解决“按钮权限对了，入口没了”的问题。

```markdown
## 权限入口可见性矩阵

| 层级 | 权限码 | 无权限行为 | 有权限行为 |
|------|--------|-----------|-----------|
| 菜单入口 | `platform.user.list` | 菜单隐藏 | 菜单显示 |
| 页面进入 | `platform.user.list` | 显示 403 | 可进入页面 |
| 查询区 | `platform.user.list` | 不可见 | 可见 |
| 新增按钮 | `platform.user.create` | 隐藏 | 可见 |
| 编辑按钮 | `platform.user.update` | 隐藏 | 可见 |
```

### 2.7 E2E 测试证据矩阵

必须先定义“打算怎么证明完成了”，不能到最后才模糊补一句“已测”。

```markdown
## E2E 测试证据矩阵

| 场景 | 测试文件 | 核心断言 | 前置数据 |
|------|---------|---------|---------|
| 页面渲染 | `platform-users.spec.ts` | 标题、列表、分页可见 | 种子管理员 |
| 权限入口 | `platform-users.spec.ts` | 有 list 权限时菜单可见 | 权限测试用户 |
| 表单验证 | `platform-users.spec.ts` | 用户名为空显示错误 | 无 |
| 标签绑定 | `platform-users.spec.ts` | `用户名` label 对应 username 输入框 | 无 |
```

### 2.8 任务快照与缓存更新要求

必须写清当前提示词要求 Agent 更新哪些文件：

```markdown
## 任务快照与缓存更新要求

1. 更新 `.ai/tasks/...` 当前任务文件中的“任务快照”区块。
2. 输出 `.ai/workspace/session-summary-*.md`。
3. 在 session summary 中记录：dxdocs 查询、验证命令、E2E 结果、剩余风险。
```

## 三、禁止写法

1. 禁止使用“等”“等等”“其余类似”。
2. 禁止只写“实现查询功能”“实现用户新增”。
3. 禁止只写“编译通过”作为验收标准。
4. 禁止省略权限入口可见性矩阵。
5. 禁止省略 E2E 证据矩阵。
6. 禁止省略 DevExtreme 能力矩阵。
7. 禁止省略“可视化设计审查包”。
8. 禁止只给连续长段文本，不给页面蓝图、交互流和覆盖矩阵。
9. 禁止把模块 Epic 直接当作一次性执行单元。

## 四、验收标准的正确写法

验收标准必须分层，且以 checklist 表达：

### P0 — 功能点完整性

- [ ] 页面标题存在
- [ ] 查询区字段完整
- [ ] 列表列完整
- [ ] 操作按钮完整
- [ ] 分页完整

### P1 — 业务规则完整性

- [ ] 唯一性校验存在
- [ ] 权限入口可见性矩阵落实
- [ ] 状态流转显隐正确
- [ ] 危险操作确认存在

### P2 — 生产质量完整性

- [ ] 组件复用决策已落地
- [ ] 可视化设计审查包已补齐且与正文一致
- [ ] DataGrid 能力矩阵全部落实
- [ ] 响应式可用
- [ ] data-testid 覆盖关键节点

### P3 — i18n 与验证

- [ ] 5 个语言文件齐全
- [ ] key 一致
- [ ] 所有静态配置文本已国际化
- [ ] E2E 证据矩阵覆盖完成

### P4 — 构建与自审

- [ ] `npm run build` 通过
- [ ] Playwright 已实际运行
- [ ] self-review F1-F7 通过
