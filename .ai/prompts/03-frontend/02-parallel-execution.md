# 前端并行执行与续接规范

> 本文件定义前端任务的并行执行策略和多轮续接协议。
> 适用于 GitHub Agents 多轮、并行、可续接执行场景。

---

## 一、并行执行规则

### 1.1 可并行的任务

以下任务之间**无依赖关系**，可由不同 Agent 并行执行：

| 并行组 | 包含的任务 | 前置条件 |
|:------:|----------|---------|
| 组 A | F2-2 用户管理、F2-3 角色管理、F2-4 权限管理 | F1-1 主布局已完成 |
| 组 B | F2-5 租户管理、F2-6 租户信息、F2-7 租户资源、F2-8 租户配置 | F1-1 主布局已完成 |
| 组 C | F2-9 套餐、F2-10 订阅、F2-11 账单 | F1-1 主布局已完成 |
| 组 D | F2-12 API 集成、F2-13 审计、F2-14 通知、F2-15 文件、F2-16 平台运营 | F1-1 主布局已完成 |

### 1.2 不可并行的任务

以下任务必须串行执行：

- F0-1（脚手架）→ F0-2/F0-3/F0-4/F0-5/F0-6（基础设施并行）→ F1-1/F1-2（布局）
- 同一模块的不同子任务（如用户列表 → 用户表单 → 用户详情）

### 1.3 并行冲突预防

多 Agent 并行时，以下文件可能产生冲突：

| 文件 | 冲突类型 | 解决方案 |
|------|---------|---------|
| `src/router/index.ts` | 路由注册 | 每个模块只追加自己的路由，不修改其他模块 |
| `src/constants/permissions.ts` | 权限码 | 按模块分段追加，不修改已有段 |
| `src/store/app.ts` | 菜单数据 | 菜单数据来自后端/配置，不在 store 中硬编码 |

### 1.4 并行输出隔离

每个 Agent 的输出文件必须在各自的模块目录下：

```
src/WebTenantPlatfrom/src/
├── views/
│   ├── platform-users/      ← Agent A 独占
│   ├── platform-roles/      ← Agent B 独占
│   ├── tenants/             ← Agent C 独占
│   └── ...
├── api/
│   ├── platform-users.ts    ← Agent A 独占
│   ├── platform-roles.ts    ← Agent B 独占
│   └── ...
└── types/
    ├── platform-users.ts    ← Agent A 独占
    ├── platform-roles.ts    ← Agent B 独占
    └── ...
```

## 二、多轮续接协议

### 2.1 每轮必须输出

每轮 Agent 执行结束时，必须输出以下内容：

#### A. 本轮执行结果

```markdown
## 本轮执行结果

### 新增文件
- [文件路径]: [用途说明]

### 删除文件
- [文件路径]: [删除原因]

### 重写文件
- [文件路径]: [变更说明]

### 保留未修改文件
- [文件路径]: [保留原因]
```

#### B. 新旧提示词映射（如果是提示词重构任务）

```markdown
## 新旧提示词映射

| 旧文件 | 处理方式 | 新文件 |
|--------|---------|--------|
| 03-frontend/scaffold.md | 替换 | 03-frontend/04-devextreme-templates.md |
| ... | ... | ... |
```

#### C. 下一轮计划

```markdown
## 下一轮计划

### 优先处理
1. [文件/模块名]: [目标说明]
2. ...

### 不需要重复阅读的稳定文件
- [文件路径]（原因：内容已稳定，本轮未修改）

### 仍需保持的规则
- [规则描述]（来源：[文件路径]）
```

#### D. 续接入口

```markdown
## 续接说明

当前已完成到：[阶段描述]
下一轮优先处理：[具体文件/任务]
新 Agent 接手需先阅读：
1. `.ai/workspace/session-summary-{YYYYMMDD}-{seq}.md`（本轮产出）
2. [其他需要阅读的文件]
```

### 2.2 Session Summary 文件

每轮结束时在 `.ai/workspace/` 下创建 session summary 文件：

**文件名格式**：`session-summary-{YYYYMMDD}-{seq}.md`

**必须包含内容**：

1. 任务目标
2. 本轮完成内容
3. 文件变更清单
4. 验收 checklist 结果
5. 乱码检查结果
6. 下一轮建议
7. 续接注意事项

### 2.3 新 Agent 接手流程

新 Agent 接手前端任务时必须：

1. **读取最新的 session summary**
2. **读取 `03-frontend/00-governance.md`（总治理文件）**
3. **确认当前已完成的层级和任务**
4. **确认下一步要执行的任务编号**
5. **验证仓库当前状态**（`git status`、`npm run build`）
6. **不重复执行已完成且已验收的任务**

### 2.4 续接上下文恢复

新 Agent 不需要重新阅读以下已稳定文件（除非 session summary 标注有修改）：

- `.ai/rules/global.md`
- `.ai/rules/frontend.md`
- `.ai/rules/i18n.md`
- `.ai/context/tech-stack.md`
- `.ai/system/agent-contract.md`

但必须重新阅读：

- 最新的 `session-summary-*.md`
- `03-frontend/00-governance.md`（可能有更新）
- 当前任务对应的业务提示词

## 三、异常处理

### 3.1 任务执行中断

如果 Agent 在任务执行中被中断：

1. **立即输出当前进度**（通过 `report_progress`）
2. **记录中断点**（哪个文件、哪个步骤）
3. **输出 partial session summary**
4. 下一轮 Agent 从中断点继续

### 3.2 编译失败

如果 `npm run build` 失败：

1. **分析错误信息**
2. **修复编译错误**
3. **重新编译**
4. 如果 3 次尝试后仍失败，输出错误详情并请求人工介入

### 3.3 并行冲突

如果多 Agent 产生文件冲突：

1. **不自行 merge**
2. **输出冲突文件列表和各自的变更内容**
3. **请求人工决定合并方式**
