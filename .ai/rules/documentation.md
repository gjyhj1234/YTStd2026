# 文档规范

## 目标

定义项目文档的编写标准、组织方式和维护规则。

---

## 适用范围

`docs/` 目录下所有文档和代码中的 XML 注释。

---

## 文档类型

| 类型 | 位置 | 说明 |
|------|------|------|
| 架构文档 | `docs/architecture.md` | 仓库整体架构 |
| 模块架构 | `docs/{Module}/architecture.md` | 模块架构说明 |
| API 文档 | `docs/{Module}/API.md` | API 接口文档 |
| 数据字典 | `docs/{Module}/database_dictionary.md` | 数据库表结构 |
| 项目参考 | `docs/existing-projects-reference.md` | 已有项目 API 参考 |
| 重建文档 | `docs/reconstruction/` | 重建过程文档 |
| 标准文档 | `docs/standards/` | 开发标准文档 |

---

## 编写规范

### 语言

- 所有文档使用中文
- 技术术语保持英文（如 API、DTO、JWT）
- 代码示例中的注释使用中文

### Markdown 格式

- 标题从 `#` 开始，按层级递增
- 标题层级不跳级（不从 `#` 直接到 `###`）
- 列表使用 `-` 而非 `*`
- 代码块指定语言（如 ````csharp`、````json`）
- 表格使用 Markdown 表格语法
- 文件间引用使用相对路径

### 标题编号

长文档使用编号标题保持结构清晰：

```markdown
## 1. 概述
## 2. 架构设计
### 2.1 部署模式
### 2.2 分层结构
## 3. API 设计
```

## API 文档格式

每个 API 端点的文档格式：

```markdown
### POST /api/platform-users

创建平台用户。

**权限码**：`platform.user.create`

**请求体**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| Username | string | ✓ | 用户名，1-50 字符 |
| DisplayName | string | ✓ | 显示名称 |
| Email | string | | 邮箱地址 |

**响应**：

| 字段 | 类型 | 说明 |
|------|------|------|
| Code | int | 0=成功 |
| Message | string | 结果消息 |
| Data.Id | long | 新用户 ID |

**错误码**：

| 错误码 | 说明 |
|-------|------|
| 18001 | 用户名已存在 |
| 12001 | 必填字段缺失 |
```

## 数据字典格式

每个表的文档格式：

```markdown
### plt_user（平台用户）

| 列名 | 类型 | 可空 | 默认值 | 说明 |
|------|------|:----:|-------|------|
| id | bigint | ✗ | — | 主键 |
| username | varchar(50) | ✗ | — | 用户名 |
| is_enabled | boolean | ✗ | true | 是否启用 |
| created_at | timestamptz | ✗ | — | 创建时间 |

**索引**：
- `uq_plt_user_username`：`username`（唯一）

**关联**：
- `plt_user_role`：用户-角色关联
```

## 文档维护规则

### 同步更新

以下操作必须同步更新文档：

| 代码变更 | 必须更新 |
|---------|---------|
| 新增/修改实体 | 数据字典 |
| 新增/修改 API | API 文档 |
| 修改架构 | 架构文档 |
| 新增已有项目 API | 项目参考文档 |

### 禁止的文档问题

- 禁止文档中出现乱码或异常字符
- 禁止标题层级不连续
- 禁止使用"应该/可以"等模糊表达定义规则
- 禁止有断裂链接
- 禁止文档内容与代码不一致

### 文档审查

每个阶段完成后，检查：

- [ ] 文档与代码一致
- [ ] 无乱码/异常字符
- [ ] 标题层级正确
- [ ] 链接有效
- [ ] 表格格式正确
- [ ] 编号连续
