# 国际化规范

## 目标

统一前端和后端的国际化职责分工、实现方式和资源管理规范。基于 YTStdI18n.Generator 实现后端与前端的强一致性国际化。

---

## 适用范围

所有涉及多语言的前端和后端代码。本规范适用于本项目及所有使用 YTStdI18n 体系的项目。

---

## 支持的语言

| 语言 | 代码 | 后端枚举 | 前端 locale | 备注 |
|------|------|---------|------------|------|
| 简体中文 | zh-CN | `Lang.ZhCn (0)` | `zh-CN` | 基准语言 |
| 英语 | en-US | `Lang.En (1)` | `en-US` | 国际化首选 |
| 日语 | ja-JP | `Lang.Ja (2)` | `ja-JP` | — |
| 繁体中文 | zh-TW | `Lang.ZhTw (3)` | `zh-TW` | — |
| 马来语 | ms-MY | `Lang.MsMy (4)` | `ms-MY` | — |

默认语言：简体中文（zh-CN）。

---

## 核心设计原则

### 后端零文本原则

- 后端不保留任何用户可见的字符串文本
- 所有错误消息、业务消息、枚举显示名、系统通知模板均使用**整形常量**表示
- 后端仅传递常量 Code（int），由前端负责翻译为对应语言文本
- Messages 常量值为 i18n key 字符串格式（dot.separated），供前端翻译时查找

### Generator 前后端强对应原则

- YTStdI18n.Generator 同时生成后端常量和前端语言包
- 后端常量与前端 key 必须一一对应
- Generator 生成前端代码时采用增量策略：已存在的 key 不修改、不存在的 key 才新增

## 前后端职责分工

### 后端职责

| 内容 | 后端负责 | 说明 |
|------|:-------:|------|
| API 错误消息 | ✓ | 使用 `Messages.XXX` 整形常量（对应 i18n key） |
| 业务规则错误 | ✓ | 使用 `Messages.XXX` 整形常量 |
| 枚举值 | ✓ | 返回整形值，不返回显示名文本 |
| 系统通知模板 | ✓ | 使用整形常量标识模板，不包含文本 |
| 日志消息 | ✗ | 日志内容不需要国际化 |
| 数据库字段值 | ✗ | 数据库存储原始值 |

### 前端职责

| 内容 | 前端负责 | 说明 |
|------|:-------:|------|
| 后端错误消息翻译 | ✓ | 接收后端 i18n key，翻译为当前语言文本显示 |
| 枚举显示名翻译 | ✓ | 接收后端枚举整形值，翻译为当前语言文本 |
| 通知模板翻译 | ✓ | 接收模板常量，翻译为当前语言文本 |
| 菜单文本 | ✓ | `t('global.menu.dashboard')` |
| 按钮文本 | ✓ | `t('global.button.save')` |
| 表单标签 | ✓ | `t('User.Form.username')` |
| 表格列标题 | ✓ | `t('User.List.columns.username')` |
| 验证消息 | ✓ | `t('global.validation.required')` |
| 功能说明 | ✓ | `t('User.List.description.title')` |
| 操作指引 | ✓ | `t('User.List.guide.step1')` |
| 日期时间格式 | ✓ | 使用 `Intl.DateTimeFormat` 或 `format/datetime.ts` |
| 数字格式 | ✓ | 使用 `Intl.NumberFormat` 或 `format/number.ts` |
| 货币格式 | ✓ | 使用 `Intl.NumberFormat` 或 `format/currency.ts` |

## 后端国际化实现

### Messages 常量

```csharp
// Application/Constants/Messages.cs
public static class Messages
{
    // 通用
    public const string Success = "common.success";
    public const string NotFound = "common.not_found";
    public const string OperationFailed = "common.operation_failed";

    // 认证
    public const string InvalidCredentials = "auth.invalid_credentials";
    public const string TokenExpired = "auth.token_expired";

    // 用户
    public const string UsernameExists = "user.username_exists";
    public const string UserNotFound = "user.not_found";
    public const string UserDisabled = "user.disabled";
}
```

### 使用方式

```csharp
// 正确 — 使用 Messages 常量
return ApiResult.Fail(ErrorCodes.UsernameExists, Messages.UsernameExists);

// 错误 — 禁止硬编码中文
return ApiResult.Fail(ErrorCodes.UsernameExists, "用户名已存在");

// 错误 — 禁止直接使用 i18n key 字符串
return ApiResult.Fail(ErrorCodes.UsernameExists, "user.username_exists");
```

### YTStdI18n 使用

```csharp
// 服务端需要翻译时（如邮件模板、通知内容）
var text = I18n.T(tenantId, LangKeys.SomeKey);
```

### I18nResourceAttribute 使用

```csharp
// 使用 I18nResourceAttribute 标注资源类，由 YTStdI18n.Generator 处理
[I18nResource]
public static class UserMessages
{
    // Generator 将生成对应的整形常量和前端语言包
}
```

## 前端目录规则（强制约束）

### 1. generated 目录（自动生成，禁止手改）

路径规则：

```text
locales/generated/{类型}/{模块}/{语言}.json
```

示例：

```text
generated/error/user/zh-CN.json
generated/enum/user_status/en-US.json
generated/message/common/zh-CN.json
generated/notification/system/zh-CN.json
```

由 YTStdI18n.Generator 根据后端 `I18nResourceAttribute` 标注的资源类自动生成。生成规则：
- 检查各语言文件中 key 是否已存在
- 已存在的 key **不修改**原有翻译对应关系
- 不存在的 key **新增**到对应语言文件

### 2. components 目录（必须与前端组件路径一致，强绑定）

路径映射规则：

```text
src/views/User/List.vue
↓
locales/components/User/List/zh-CN.json
```

嵌套组件规则：

```text
src/views/Order/Detail/Items.vue
↓
locales/components/Order/Detail/Items/zh-CN.json
```

通用组件规则：

```text
src/components/Dialog.vue
↓
locales/components/Common/Dialog/zh-CN.json
```

### 3. global 目录（非组件绑定的全局资源）

用于：菜单、按钮、校验、指引、描述等全局性文本。

```text
locales/global/{模块}/{语言}.json
```

### 4. format 目录（格式化工具）

用于：日期时间、数字、货币等格式化函数。

```text
locales/format/datetime.ts
locales/format/number.ts
locales/format/currency.ts
```

## 语言文件规则（强制）

### 每个目录必须包含的语言文件

- `zh-CN.json`（必须 — 基准语言）
- `en-US.json`（必须 — 国际化首选）
- 其他语言按需添加

### 禁止行为

- 禁止缺少 zh-CN 文件
- 禁止同一目录下不同语言的 key 集合不一致
- 禁止跨组件复用 key（每个组件有独立的 key 空间）
- 禁止手动修改 generated 目录下的文件

---

## 自动校验规则（必须实现）

### 1. 组件一致性校验

```text
src/views/User/List.vue
必须存在：
locales/components/User/List/zh-CN.json
```

### 2. key 使用检测

- 组件中使用了 `t('User.List.xxx')` 但 key 不存在 → 报错
- 语言文件中定义了 key 但组件中未使用 → 标记警告

### 3. 多语言一致性校验

```text
zh-CN.json 与 en-US.json 的 key 必须完全一致
```

### 4. 空文件处理

```text
如果 json 文件为空 → 删除该文件
```

## 加载与合并机制

### 自动加载范围

```text
locales/
├── generated/**      # 后端生成的语言包
├── components/**     # 组件级语言包
├── global/**         # 全局语言包
```

### 合并优先级顺序（从低到高）

```text
1. generated（最低优先级）
2. global
3. components（最高优先级）
```

当 key 冲突时，高优先级覆盖低优先级。

### 合并实现

```typescript
// locales/merge.ts
export function mergeLocales() {
  return deepMerge(
    generatedLocales,
    globalLocales,
    componentLocales
  );
}
```

## 组件开发强制流程

新增组件时必须同步执行以下步骤：

1. **创建组件**：`src/views/Product/List.vue`
2. **创建语言目录**：`locales/components/Product/List/`
3. **创建语言文件**：`zh-CN.json` + `en-US.json`
4. **在组件中使用 `t()` 引用 key**

禁止跳过任何步骤。禁止组件中出现硬编码中文。

## YTStdI18n.Generator 与前端协作规则

### Generator 负责范围（仅限 generated 目录）

- 读取后端 `I18nResourceAttribute` 标注的资源
- 生成 `locales/generated/` 下对应的语言文件
- 将后端整形常量 Code 映射为前端 i18n key
- 自动删除后端已废弃的 key（后端不再定义则从 generated 删除）
- **增量生成策略**：已存在的 key 不修改翻译内容，仅新增不存在的 key

### 前端负责范围（components + global + format）

- `components/` — 组件级国际化文本（手写维护）
- `global/` — 全局资源（菜单、按钮、校验、指引、描述）
- `format/` — 日期、数字、货币格式化

### 禁止交叉

- 前端禁止手动修改 `generated/` 目录
- Generator 禁止生成到 `components/`、`global/`、`format/` 目录

## generated 语言包分文件规则

后端的错误、消息、枚举、通知模板按模块生成到独立文件：

```text
generated/
├── error/           # 错误消息
│   ├── user/        # 用户模块错误
│   │   ├── zh-CN.json
│   │   └── en-US.json
│   ├── auth/        # 认证模块错误
│   ├── tenant/      # 租户模块错误
│   └── ...
├── message/         # 业务消息
│   ├── common/      # 通用消息
│   └── ...
├── enum/            # 枚举显示名
│   ├── user_status/
│   ├── tenant_status/
│   └── ...
└── notification/    # 通知模板
    ├── system/
    └── ...
```

每个模块生成独立文件夹，每种语言一个 JSON 文件。禁止将所有内容合并到单个文件中。

## 租户与用户语言配置

### 语言优先级（从高到低）

1. 用户个人语言设置
2. 租户默认语言设置
3. 系统默认语言（zh-CN）

### 前端实现

```typescript
// 获取用户语言偏好
function getUserLocale(): string {
  const userLang = currentUser.language;   // 用户个人设置
  const tenantLang = currentTenant.language; // 租户默认设置
  const systemLang = 'zh-CN';               // 系统默认
  return userLang || tenantLang || systemLang;
}
```

### 语言切换

- 用户可在个人设置中切换语言
- 切换后立即生效，无需刷新页面
- 语言选择保存到用户配置

## 回退策略

### 前端回退

1. 查找当前 locale 的翻译
2. 如果未找到，回退到 `zh-CN`（基准语言）
3. 如果仍未找到，显示 key 本身

### 后端回退

1. 查找租户配置的语言翻译
2. 如果未找到，回退到简体中文（`Lang.ZhCn`）
3. 如果仍未找到，返回空字符串

## 不需要国际化的内容

| 内容 | 原因 |
|------|------|
| 代码注释 | 面向开发者，统一使用中文 |
| 日志消息 | 面向运维，不需要多语言 |
| 数据库字段名 | 技术标识 |
| API 路径 | 技术标识 |
| 错误码数字 | 技术标识 |
| 配置键 | 技术标识 |
| 内部常量 | 技术标识 |

## DevExtreme 组件本地化

DevExtreme 有独立的 locale 机制：

```typescript
import { locale, loadMessages } from 'devextreme/localization';
import zhMessages from 'devextreme/localization/messages/zh';

loadMessages(zhMessages);
locale('zh');
```

DevExtreme 的组件内置文本（如分页器、过滤器、排序提示）通过 DevExtreme 自身的 locale 机制处理，不通过 vue-i18n。

## AI 开发时的 i18n 检查

Agent 在开发时必须确保：

1. 所有 `ApiResult.Fail()` 的 message 参数使用 `Messages.XXX` 常量
2. 所有前端用户可见文本使用 `$t()` 或 `t()`
3. 新增组件时同步创建 `locales/components/{Module}/{Component}/zh-CN.json` 和 `en-US.json`
4. 所有新增的 i18n key 在基准语言文件中有对应条目
5. 所有新增的错误消息在 `Messages.cs` 中有对应常量
6. 新增模块时同步更新所有语言的资源文件
7. 禁止手动修改 `locales/generated/` 目录下的任何文件
8. generated 目录由 `dotnet build` 触发 YTStdI18n.Generator 自动更新

## 路径映射关系速查

```text
后端错误常量 → 前端翻译:
  Messages.UsernameExists ("user.username_exists")
  → locales/generated/error/user/zh-CN.json → { "user_not_exist": "用户不存在" }
  → t('error.user.username_exists')

后端枚举值 → 前端翻译:
  UserStatus.Active (0)
  → locales/generated/enum/user_status/zh-CN.json → { "0": "启用" }
  → t('enum.user_status.0')

组件路径 → 语言路径:
  src/views/User/List.vue
  → locales/components/User/List/zh-CN.json
  → key 前缀: User.List
  → t('User.List.title')
```
