# 国际化规范

## 目标

统一前端和后端的国际化职责分工、实现方式和资源管理规范。基于 YTStdI18n.Generator 实现后端与前端的强一致性国际化。

---

## 适用范围

所有涉及多语言的前端和后端代码。本规范适用于本项目及所有使用 YTStdI18n 体系的项目。

### 前端项目路径

| 项目 | 路径 | 国际化方案 | 状态 |
|------|------|----------|------|
| 新前端 | `src/WebTenantPlatfrom` | vue-i18n + import.meta.glob 自动加载 | 活跃开发 |
| 旧前端 | `web/tenant-platform-web` | 自定义 t()/gt()（`locales/runtime/t.ts`） | 已冻结 |

> 本规范中的前端路径示例优先使用新前端路径 `src/WebTenantPlatfrom/src/locales/`。

## 一、核心定义（必须统一认知）

### 1. error（错误）

- **一定存在**
- 表示：业务错误、异常
- 表现形式：**整形 Code**

```csharp
// Application/Constants/ErrorCodes.cs
public const int UserNotExist = 100001;
```

### 2. message（提示）

- **可选存在**
- 表示：提示信息（如确认框、成功提示等）
- 同样使用：**整形 Code**

```csharp
// Application/Constants/MessageCodes.cs
public const int ConfirmDelete = 200001;
```

### 关键结论

| 类型 | 是否必须 | 表现形式 |
|------|:--------:|:--------:|
| error | ✅ 必须 | int |
| message | ❌ 可选 | int |

## 二、后端规范

### 后端零文本原则

- 后端不保留任何用户可见的字符串文本
- 所有错误和提示均使用**整形常量** Code 表示
- 后端仅传递整形常量 Code（int），由前端根据 Code 翻译为对应语言文本
- **后端不存在 Messages 类**（原有 Messages 概念已移除，统一使用 ErrorCodes）

### ApiResult.Fail 使用方式（最终规范）

```csharp
// ✅ 正确 — 仅传 ErrorCode，不传 message 参数
return ApiResult.Fail(ErrorCodes.UserNotExist);

// ❌ 禁止 — 不再允许 message 参数
return ApiResult.Fail(ErrorCodes.UserNotExist, xxx);

// ❌ 禁止 — 禁止硬编码中文
return ApiResult.Fail(ErrorCodes.UserNotExist, "用户不存在");

// ❌ 禁止 — 禁止使用字符串常量
return ApiResult.Fail(ErrorCodes.UserNotExist, "user.not_exist");
```

### 枚举值

```csharp
// ✅ 正确 — 枚举只有整形值
public enum UserStatus
{
    Active = 0,
    Disabled = 1,
    Locked = 2
}
// 前端通过 t(code.toString()) 获取显示文本

// ❌ 错误 — 枚举附带显示名
[Description("启用")]
Active = 0
```

### YTStdI18n 使用

```csharp
// 服务端需要翻译时（如邮件模板、通知内容）
var text = I18n.T(tenantId, LangKeys.SomeKey);
```

### I18nResourceAttribute 使用

```csharp
// 使用 I18nResourceAttribute 标注资源类，由 YTStdI18n.Generator 处理
[I18nResource]
public static class UserMessages
{
    // Generator 将生成对应的整形常量和前端语言包
}
```

### 目录说明

| 目录 | 用途 | 维护方式 | 优先级 |
|------|------|---------|:------:|
| `{Component}.vue.{locale}.json` | 组件级翻译 | 手写，紧贴 .vue 文件 | 最高 |
| `locales/{locale}.json` | 应用级结构化翻译（app/menu/route） | 手写 | 高 |
| `locales/enums/` | 枚举值翻译 | 手写 | 中 |
| `locales/generated/` | 后端 Code → 文本映射 | Generator 自动生成 | 中低 |
| `locales/common/` | 全局复用文本 | 手写 | 最低 |

## 五、组件级语言文件自动加载机制（核心实现）

### 实现原理

使用 Vite 的 `import.meta.glob` 在编译时自动收集所有 `*.vue.{locale}.json` 文件，过滤掉 null 值后合并到 vue-i18n 全局消息中，以最高优先级覆盖。

### locales/index.ts 关键代码（可复用到其他业务工程）

```typescript
// ① 使用 import.meta.glob 自动加载所有组件级语言文件
//    eager: true — 编译时静态导入
//    import: 'default' — 直接获取 JSON 默认导出
const componentZhCNModules = import.meta.glob<Record<string, string | null>>(
  ['../views/**/*.vue.zh-CN.json', '../components/**/*.vue.zh-CN.json', '../layouts/**/*.vue.zh-CN.json'],
  { eager: true, import: 'default' },
)
// ... 对每种语言（en-US、ms-MY、zh-TW、ja-JP）各一组 glob

// ② 合并组件级消息：仅保留非 null 值
//    null 值表示"此 key 由 common 层提供翻译"
function mergeComponentMessages(modules: Record<string, Record<string, string | null>>): Record<string, string> {
  const result: Record<string, string> = {}
  for (const data of Object.values(modules)) {
    if (!data) continue
    for (const key of Object.keys(data)) {
      const value = data[key]
      if (value !== null && value !== undefined) {
        result[key] = value
      }
    }
  }
  return result
}

// ③ 消息合并优先级（从低到高）：
//    common → generated → enums → 主语言文件 → 组件级语言文件
const messages = {
  'zh-CN': { ...commonZhCN, ...generatedZhCN, ...enumsZhCN, ...zhCN, ...componentZhCN },
  // ...
}
```

### 解析优先级

| 层级 | 说明 | 优先级 |
|------|------|:------:|
| 组件 | 当前组件的 `.vue.{locale}.json`（非 null 值） | 最高 |
| 主语言文件 | `locales/{locale}.json`（嵌套结构键） | 高 |
| enums | `locales/enums/{locale}.json` | 中 |
| generated | `locales/generated/{locale}.json` (Code→文本) | 中低 |
| common | `locales/common/{locale}.json` | 最低 |
| fallback | key 本身 | 兜底 |

## 六、组件语言文件规范

### 文件位置

组件语言文件必须紧贴对应的 `.vue` 文件：

```text
src/views/User/User.vue              → src/views/User/User.vue.zh-CN.json
src/views/User/User.vue              → src/views/User/User.vue.en-US.json
src/components/Dialog/Dialog.vue     → src/components/Dialog/Dialog.vue.zh-CN.json
```

### 组件 JSON 结构示例

```json
{
  "用户列表": "用户列表",
  "用户名": null,
  "状态": null,
  "保存": null
}
```

### null 的意义（关键设计）

```text
null = 显式声明"使用全局 common"
```

- 值为具体文本：组件自有翻译
- 值为 `null`：从 `common` 中获取翻译
- key 不存在：进入 fallback 链

### 对应 common

```json
{
  "用户名": "User Name",
  "状态": "Status",
  "保存": "Save"
}
```

### 优势

- 无需在 JSON 中写 `gt()`
- JSON 纯净、可静态分析
- 可校验 null 对应 common 是否存在

## 七、Generator 生成规范

### 输出格式

Generator 将后端整形 Code 映射为前端可翻译文本：

```json
{
  "100001": "用户不存在",
  "100002": "用户已禁用",
  "200001": "确认删除"
}
```

### 使用方式

```typescript
// 后端返回 errorCode，前端用 t() 翻译
t(errorCode.toString())
```

### 增量生成策略

- 遍历后端定义的所有整形 Code
- 检查前端对应语言文件中该 Code 是否已存在
- **已存在 → 不修改**（保留人工翻译校正）
- **不存在 → 新增**（使用默认翻译值）
- **后端已删除 → 从 generated 文件中删除**

### 生成目标

```text
web/{project}/src/locales/generated/
├── zh-CN.json    # { "100001": "用户不存在", ... }
└── en-US.json    # { "100001": "User not found", ... }
```

## 八、校验机制

### 1. null 校验

如果组件 JSON 中 key 的值为 `null`，则该 key **必须**在 `common` 中存在。否则产生 warning。

### 2. 缺失 key 检测

- 组件中使用了 `t('xxx')` 但 key 在组件 JSON 和 common 中都不存在 → warn
- 语言文件中定义了 key 但组件中未使用 → 标记警告

### 3. 多语言一致性

- 同一组件的 `.zh-CN.json` 和 `.en-US.json` 的 key 必须完全一致
- `common/zh-CN.json` 和 `common/en-US.json` 的 key 必须完全一致
- `generated/zh-CN.json` 和 `generated/en-US.json` 的 key 必须完全一致

## 十、语言文件规则（强制）

### 每个目录/组件必须包含的语言文件

- `*.vue.zh-CN.json`（必须 — 基准语言）

- `*.vue.ms-MY.json`（必须）
- `*.vue.zh-TW.json`（必须）
- `*.vue.ja-JP.json`（必须）

### 禁止行为

- 禁止缺少 zh-CN 文件
- 禁止同一组件下不同语言的 key 集合不一致
- 禁止使用 dot.separated 格式的 key（使用中文作为 key）
- 禁止在组件 JSON 中写 `gt('xxx')`（使用 `null` 代替）
- **禁止将组件特有的扁平中文 key 放在主语言文件（`locales/{locale}.json`）中**

## 十一、组件开发强制流程

新增组件时必须同步执行以下步骤：

1. **创建组件**：`src/views/User/UserView.vue`
2. **创建 5 个语言文件**：`UserView.vue.zh-CN.json` + `UserView.vue.en-US.json` + `UserView.vue.ms-MY.json` + `UserView.vue.zh-TW.json` + `UserView.vue.ja-JP.json`（紧贴 .vue 文件）
3. **在 JSON 中声明 key**：自有文本写具体翻译值，复用文本写 `null`
4. **在组件中使用 `$t()` 引用 key**
5. **确认 `locales/index.ts` 的 `import.meta.glob` 能匹配到新文件**（glob 路径包含 `views/**`、`components/**`、`layouts/**`）

禁止跳过任何步骤。禁止组件中出现硬编码中文。
禁止将组件特有的翻译 key 写入主语言文件 `locales/{locale}.json`——`import.meta.glob` 自动加载组件级文件。

## 十二、Generator 与前端协作规则

### Generator 负责范围（仅限 generated 目录）

- 读取后端 `I18nResourceAttribute` 标注的资源
- 生成 `locales/generated/{locale}.json`
- 将后端整形 Code 映射为前端可翻译文本
- 自动删除后端已废弃的 Code
- **增量策略**：已存在的 key 不修改翻译内容，仅新增不存在的 key

### 前端负责范围

- 组件级 `.vue.{locale}.json` — 组件自有翻译（手写维护）
- `locales/common/` — 全局复用文本（手写维护）
- `locales/runtime/` — t()/gt()/loader 实现

### 禁止交叉

- generated 目录的文件由 Generator 自动生成和管理新增 key，翻译内容允许手动校正编辑
- Generator 禁止生成到 `common/`、`runtime/` 目录
- Generator 禁止生成到组件目录

## 十四、回退策略

### 前端回退（通过 vue-i18n 消息合并优先级实现）

消息合并后，vue-i18n 的 `$t()` / `t()` 直接查找全局消息对象。优先级由合并顺序决定（后合并的覆盖先合并的）：

1. **组件级语言文件**（最高优先级）— `*.vue.{locale}.json` 中非 null 的值
2. **主语言文件** — `locales/{locale}.json` 中的嵌套结构键
3. **enums** — `locales/enums/{locale}.json`
4. **generated** — `locales/generated/{locale}.json`（Code→文本）
5. **common** — `locales/common/{locale}.json`（最低优先级）
6. **fallback** — key 本身（即中文原文）

组件 JSON 中值为 `null` 的 key 不会被 `mergeComponentMessages` 收集，因此该 key 的翻译由更低优先级的层提供。

### 后端回退

1. 查找租户配置的语言翻译
2. 如果未找到，回退到简体中文（`Lang.ZhCn`）
3. 如果仍未找到，返回空字符串

## 十六、DevExtreme 组件本地化

DevExtreme 有独立的 locale 机制：

```typescript
import { locale, loadMessages } from 'devextreme/localization';
import zhMessages from 'devextreme/localization/messages/zh';

```

DevExtreme 的组件内置文本（如分页器、过滤器、排序提示）通过 DevExtreme 自身的 locale 机制处理，不通过 `t()`。

## 十七、AI 开发时的 i18n 检查

Agent 在开发时必须确保：

1. 所有 `ApiResult.Fail()` 仅传 `ErrorCodes.XXX`，不传 message 参数
2. 所有前端用户可见文本使用 `$t()` 或 `t()`，key 为中文
3. 新增组件时同步创建 5 个语言文件（`.vue.zh-CN.json` + `.vue.en-US.json` + `.vue.ms-MY.json` + `.vue.zh-TW.json` + `.vue.ja-JP.json`）
4. 组件 JSON 中复用文本使用 `null`，自有文本写具体翻译值
5. `null` 对应的 key 在 `common/` 中必须存在
6. generated 目录允许手动编辑翻译内容（校正翻译），但 Generator 运行时不会覆盖已有 key 的值
7. generated 目录由 `dotnet build` 触发 YTStdI18n.Generator 自动新增 key
8. **组件特有的扁平中文 key 必须放在组件级语言文件中，禁止放在主语言文件 `locales/{locale}.json`**
9. **确认 `locales/index.ts` 的 `import.meta.glob` 能匹配到所有组件级文件**
