# 执行策略与行为边界

## 目标

定义 Agent 执行任务时的策略、流程与行为边界，确保每次执行都遵循"分析 → 计划 → 实现 → 验证 → 收尾"的闭环。

---

## 适用范围

所有由 Agent 执行的编码、测试、文档任务。

---

## 执行闭环

每个任务必须按以下五步执行：

### 步骤 1：分析

- 阅读任务描述与关联文件
- 理解当前代码库状态
- 识别任务边界与风险
- 确认前置条件是否满足

### 步骤 2：计划

- 列出将要创建或修改的文件清单
- 列出将要使用的技术方案
- 预估任务复杂度
- 如果任务过大，按 `.ai/system/task-splitting.md` 拆分

### 步骤 3：实现

- 按计划逐步实现
- 每完成一个逻辑单元，立即验证编译
- 遵守所有适用的规则文件
- 不超出任务边界

### 步骤 4：验证

- 执行构建验证：`dotnet build YTStd.slnx`（后端）/ `npm run build`（前端）
- 执行测试验证：`dotnet test YTStd.slnx`（后端）/ `npm run test`（前端，如有）
- 检查输出是否满足任务的验收标准
- 检查是否引入了新的编译警告或错误

### 步骤 5：收尾

- 输出会话总结
- 标记已完成和未完成的内容
- 记录风险点和待确认事项
- 给出下一步建议

---

## 行为边界

### 必须做的事情

1. 所有公开类型、方法、DTO 必须有中文 XML 注释
2. 所有 `Logger.Debug` 调用必须使用 `Func<string>` 延迟求值重载
3. 所有 `InsertAsync` 调用前必须通过 `DB.GetNextLongIdAsync()` 显式获取 ID
4. 所有错误消息必须使用 `Messages.XXX` 整形常量（`const int`）
5. 每个后端接口模块完成时，必须同步生成对应的 Postman 测试脚本
6. 每个前端模块必须包含功能说明与操作指引
7. 每次修改后必须验证编译通过

### 禁止做的事情

1. 禁止使用反射（`System.Reflection`）
2. 禁止使用 `dynamic`
3. 禁止使用 `System.Linq`
4. 禁止使用 Expression Tree
5. 禁止引入 Redis 或分布式缓存
6. 禁止拆分为微服务
7. 禁止使用 MVC 反射式 Controller
8. 禁止在租户平台实体中使用裸 `TenantId` / `tenant_id` 字段名
9. 禁止手写一次性 SQL 文件作为初始化主实现
10. 禁止删除或修改已有测试用例（除非任务明确要求）
11. 禁止使用字符串拼接构造 JSON（必须使用 `Utf8JsonWriter` 或 `JsonSerializerContext`）

## 超时处理

当 Agent 预判任务无法在单次执行内完成时：

1. 优先完成当前正在进行的最小逻辑单元
2. 确保已完成的代码可编译
3. 输出会话总结，标明已完成和未完成的内容
4. 给出下一轮的续接指令

---

## 错误恢复

当 Agent 遇到编译错误或测试失败时：

1. 首先尝试修复，最多尝试 3 次
2. 如果修复失败，回滚到最后一次可编译状态
3. 在会话总结中详细记录错误信息和回滚情况
4. 标记需要人类介入的问题
