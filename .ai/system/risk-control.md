# 风险控制规范

## 目标

定义任务执行中的风险识别、评估和控制策略，确保 Agent 以最小风险推进任务。

---

## 适用范围

所有由 Agent 执行的任务。

---

## 风险分级

### 高风险操作

以下操作在执行前必须获得人类确认：

| 操作 | 风险说明 | 控制措施 |
|------|---------|---------|
| 修改已有实体结构 | 可能影响数据库表结构和现有数据 | 人类确认后执行 |
| 删除文件 | 不可逆操作 | 人类确认后执行 |
| 修改认证/权限逻辑 | 可能导致安全漏洞 | 人类确认后执行，必须有测试 |
| 新增外部依赖 | 可能引入安全漏洞或兼容性问题 | 人类确认后执行 |
| 修改数据库初始化数据 | 可能影响生产环境 | 人类确认后执行 |
| 修改全局中间件管道 | 可能影响所有请求处理 | 人类确认后执行 |

### 中风险操作

以下操作需要 Agent 自行评估并在会话总结中报告：

| 操作 | 风险说明 | 控制措施 |
|------|---------|---------|
| 新增 API 端点 | 可能暴露未授权的数据 | 确保权限码正确配置 |
| 修改 DTO 结构 | 可能破坏前后端契约 | 检查前端是否需要同步修改 |
| 修改缓存逻辑 | 可能导致数据不一致 | 确保失效策略正确 |
| 批量修改文件 | 可能引入不可预期的副作用 | 每次修改后编译验证 |

### 低风险操作

以下操作 Agent 可自行执行：

- 新增文件（不修改已有文件）
- 添加注释
- 修复编译错误
- 添加测试
- 更新文档

---

## 回滚策略

### 编译失败回滚

当修改导致编译失败且无法在 3 次尝试内修复时：

1. 使用 `git checkout -- {file}` 回滚修改的文件
2. 记录失败原因
3. 在会话总结中标记需要人类介入

### 测试失败回滚

当修改导致已有测试失败时：

1. 分析是否为预期变化
2. 如果是预期变化，说明原因并更新测试（需人类确认）
3. 如果是意外变化，回滚修改

### 大规模修改回滚

当修改涉及超过 20 个文件且出现问题时：

1. 使用 `git stash` 保存当前修改
2. 回到最后一次稳定状态
3. 重新评估修改策略
4. 采用更小粒度的方式逐步修改

---

## 变更影响分析

在执行修改前，Agent 必须评估以下影响：

### 修改实体时

- 检查哪些服务依赖该实体
- 检查哪些 DTO 需要同步修改
- 检查哪些测试可能受影响
- 检查初始化数据是否需要更新
- 检查前端类型定义是否需要同步

### 修改 API 时

- 检查前端是否有调用该 API
- 检查 Postman 集合是否需要更新
- 检查 API 文档是否需要更新
- 检查权限配置是否需要更新

### 修改前端时

- 检查路由配置是否受影响
- 检查权限常量是否需要更新
- 检查 i18n 资源是否需要更新
- 检查其他页面是否有依赖

---

## 安全红线

以下行为绝对禁止，无论任何情况：

1. 禁止在代码中硬编码密码、密钥、Token
2. 禁止暴露数据库连接字符串
3. 禁止跳过认证检查
4. 禁止禁用 HTTPS
5. 禁止在日志中输出敏感信息（密码、Token）
6. 禁止使用 SQL 拼接（必须使用参数化查询）
7. 禁止信任客户端输入（必须做服务端验证）
8. 禁止在错误消息中暴露系统内部信息

---

## 风险报告格式

```markdown
## 风险报告

| 风险项 | 等级 | 描述 | 当前状态 | 建议措施 |
|-------|------|------|---------|---------|
| {风险名} | 高/中/低 | {描述} | 已控制/需关注/需处理 | {措施} |
```
