# 提示词编写指南

## 目标

定义提示词文件的标准结构和编写规范，确保所有提示词可被 AI 准确理解和执行。

---

## 适用范围

所有 `.ai/prompts/` 下的提示词文件。

---

## 标准结构

每个提示词文件必须包含以下章节：

```markdown
# {提示词标题}

## 目标
{一句话描述本提示词的目标}

## 适用范围
{说明本提示词适用于哪些场景}

## 前置阅读
{列出执行前必须阅读的文件}

## 输入
{描述执行本任务需要的输入}

## 输出
{描述本任务的预期产出}

## 执行步骤
{按顺序列出具体步骤}

## 约束
{列出必须遵守的规则}

## 禁止事项
{列出明确禁止的行为}

## 验收标准
{列出可检查的完成标准}

## 续接说明
{说明如何在下一轮继续}
```

---

## 编写原则

### 1. 精确性

- API 签名、类型名称必须精确无歧义
- 使用"必须"表示强制要求
- 使用"禁止"表示绝对禁止
- 使用"建议"表示推荐但不强制
- 避免使用"应该""可以"等模糊词

### 2. 可执行性

- 每个步骤必须可由 AI 直接执行
- 步骤之间有明确的顺序和依赖关系
- 每个步骤有清晰的完成判断标准
- 避免"根据实际情况决定"这类模糊指令

### 3. 完整性

- 不使用"以下略""后续同理""其余类似"
- 不使用"请自行补充"
- 所有需要列举的内容必须完整列出
- 所有需要定义的规则必须明确定义

### 4. 一致性

- 同一概念在不同提示词中使用相同术语
- 命名规则与 `.ai/rules/naming.md` 一致
- 格式规范与 `.ai/system/output-format.md` 一致

## 语言规范

| 表达 | 用法 | 示例 |
|------|------|------|
| 必须 | 强制要求，不可违反 | 所有实体必须有 `Id` 属性 |
| 禁止 | 绝对不允许 | 禁止使用反射 |
| 建议 | 推荐做法，有合理理由可偏离 | 建议使用 `ReadOnlySpan<char>` |
| 优先 | 首选方案，次选方案可接受 | 优先使用 `Func<string>` 重载 |

避免使用的表达：

| 避免使用 | 原因 |
|---------|------|
| 应该 | 语义不明确，不清楚是必须还是建议 |
| 可以 | 过于宽泛，不利于 AI 做决策 |
| 适当 | 没有具体标准 |
| 酌情 | AI 无法判断"情"是什么 |

## 引用规范

### 引用规则文件

```markdown
## 前置阅读
- `.ai/rules/global.md` — 全局开发规范
- `.ai/rules/backend.md` — 后端开发规范
```

### 引用上下文文件

```markdown
## 前置阅读
- `.ai/context/tech-stack.md` — 技术栈约束
- `docs/TenantPlatform/database_dictionary.md` — 数据字典
```

### 引用已有项目

```markdown
## 依赖项目
- `docs/existing-projects-reference.md` — 已有项目 API 参考
```

## 提示词分类

| 类别 | 目录 | 特点 |
|------|------|------|
| 治理类 | `00-governance/` | 描述流程和阶段，不涉及具体编码 |
| 项目类 | `01-project/` | 初始化和脚手架操作 |
| 实现类 | `02-backend/` ~ `06-generator/` | 具体编码任务 |
| 测试类 | `07-testing/` | 测试编写任务 |
| 平台类 | `08-platform/` | 平台模块的完整实现（含实体+服务+API+测试） |
| 文档类 | `09-docs/` | 文档生成任务 |
| 审查类 | `10-review/` | 审查和优化任务 |

## 版本管理

- 每次修改提示词时，在文件末尾更新版本号和日期
- 重大变更需要在 `.ai/README.md` 中记录
- 旧版本不删除，通过版本号区分
