# 任务拆分规范

## 目标

定义大任务的拆分策略，确保每个子任务可在单次 Agent 执行内完成，并支持多轮续接。

---

## 适用范围

所有预计执行时间超过 30 分钟或涉及 10 个以上文件的任务。

---

## 拆分原则

### 1. 单一职责

每个子任务只做一件事。一个子任务不应同时包含实体建模和 API 实现。

### 2. 可独立验证

每个子任务完成后，必须能独立通过编译和测试验证。

### 3. 有明确边界

每个子任务必须有清晰的输入、输出和验收标准。

### 4. 有依赖说明

如果子任务依赖前序子任务的输出，必须明确说明依赖关系。

### 5. 粒度适中

每个子任务的目标文件数控制在 5-15 个之间，执行时间控制在 20-50 分钟。

---

## 后端任务拆分维度

后端大任务按以下维度拆分，每个维度可独立执行：

| 拆分维度 | 子任务示例 | 预估文件数 |
|---------|----------|-----------|
| 按实体 | 创建 PlatformUser 实体及相关枚举 | 2-5 |
| 按层次 | 实现 PlatformUser 应用服务 | 3-6 |
| 按接口 | 实现 `/api/platform-users` 端点组 | 2-4 |
| 按查询与命令 | 实现 PlatformUser 的 CRUD DTO | 4-8 |
| 按验证 | 实现 PlatformUser 的输入验证 | 2-3 |
| 按权限 | 实现 PlatformUser 的权限点与中间件 | 2-4 |
| 按审计 | 实现 PlatformUser 的审计记录 | 1-3 |
| 按批量操作 | 实现 PlatformUser 的批量启用/禁用 | 1-2 |
| 按状态流转 | 实现 Tenant 的状态机 | 2-4 |
| 按测试 | 编写 PlatformUser 的单元/集成测试 | 2-5 |
| 按 Postman | 生成 PlatformUser 的 Postman 测试集合 | 1 |

### 后端典型拆分方案（以单个业务模块为例）

```
子任务 1：实体与枚举建模
  输入：database_dictionary.md 中的表定义
  输出：entity/*.cs, Domain/Enums/*.cs
  验证：dotnet build

子任务 2：DTO 与请求/响应模型
  输入：已完成的实体
  输出：Application/Dtos/{Module}/*.cs
  验证：dotnet build

子任务 3：应用服务
  输入：已完成的实体与 DTO
  输出：Application/Services/{Module}AppService.cs
  验证：dotnet build

子任务 4：API 端点
  输入：已完成的应用服务
  输出：Endpoints/{Module}Endpoints.cs
  验证：dotnet build

子任务 5：初始化数据
  输入：已完成的实体与服务
  输出：Infrastructure/Initialization/Contributors/{Module}SeedContributor.cs
  验证：dotnet build + dotnet test

子任务 6：测试
  输入：已完成的全部代码
  输出：tests/{Module}Tests.cs
  验证：dotnet test

子任务 7：Postman 测试集合
  输入：已完成的 API 端点
  输出：docs/{Module}-postman-collection.json
  验证：人工导入 Postman 验证
```

## 前端任务拆分维度

前端大任务按以下维度拆分：

| 拆分维度 | 子任务示例 | 预估文件数 |
|---------|----------|-----------|
| 按模块 | 实现平台用户管理模块 | 5-10 |
| 按页面 | 实现用户列表页 | 2-4 |
| 按组件 | 实现通用状态标签组件 | 1-2 |
| 按表单 | 实现用户创建/编辑表单 | 2-3 |
| 按表格 | 实现用户数据表格 | 1-3 |
| 按详情页 | 实现用户详情抽屉 | 2-3 |
| 按接口联调 | 对接用户管理 API | 2-4 |
| 按权限渲染 | 实现用户模块权限控制 | 1-2 |
| 按路由与菜单 | 注册用户模块路由 | 1-2 |
| 按国际化 | 接入用户模块 i18n | 2-3 |
| 按测试 | 编写用户模块测试 | 2-4 |

### 前端典型拆分方案（以单个业务模块为例）

```
子任务 1：路由与菜单注册
  输入：模块需求描述
  输出：router/index.ts 更新, constants/permissions.ts 更新
  验证：npm run build

子任务 2：API 封装
  输入：后端 API 文档
  输出：api/{module}.ts, types/{module}.ts
  验证：npm run build

子任务 3：列表页
  输入：已完成的 API 封装
  输出：views/{module}/ListView.vue
  验证：npm run build

子任务 4：表单与编辑
  输入：已完成的列表页
  输出：views/{module}/CreateView.vue, EditView.vue
  验证：npm run build

子任务 5：详情页
  输入：已完成的 API 封装
  输出：views/{module}/DetailView.vue
  验证：npm run build

子任务 6：功能说明与操作指引
  输入：模块业务需求
  输出：功能说明卡片内容, 操作指引内容
  验证：npm run build

子任务 7：国际化接入
  输入：已完成的全部页面
  输出：locales/zh-CN.json 更新, locales/en-US.json 更新
  验证：npm run build
```

## 平台模块拆分

平台基础模块按以下顺序拆分执行：

```
第 1 批：认证与权限
  - 认证中间件
  - 权限模型与缓存
  - 登录 API

第 2 批：用户与角色
  - 平台用户 CRUD
  - 平台角色 CRUD
  - 权限分配

第 3 批：租户管理
  - 租户生命周期
  - 租户信息
  - 租户资源

第 4 批：配置与字典
  - 系统配置
  - 功能开关
  - 字典数据

第 5 批：运营与审计
  - 审计日志
  - 操作日志
  - 监控统计

第 6 批：扩展能力
  - 通知系统
  - API 集成
  - 文件存储
```

---

## 数据库设计拆分

数据库相关任务按以下维度拆分：

```
子任务 1：核心表设计（平台管理 + 认证）
子任务 2：租户相关表设计
子任务 3：SaaS 业务表设计（套餐/订阅/计费）
子任务 4：运营支撑表设计（日志/通知/存储）
子任务 5：初始化数据编写
子任务 6：索引与性能优化
```

---

## 测试拆分

测试任务按以下维度拆分：

```
子任务 1：实体与生成器测试
子任务 2：初始化数据幂等性测试
子任务 3：中间件行为测试
子任务 4：核心 API 测试
子任务 5：扩展 API 测试
子任务 6：权限与安全测试
子任务 7：Postman 集合生成
```

---

## 拆分结果格式

每次拆分后，输出格式为：

```markdown
## 任务拆分结果

### 原始任务
- 描述：...
- 预估复杂度：高

### 子任务清单

| 序号 | 子任务名称 | 依赖 | 预估文件数 | 预估时间 | 状态 |
|-----|-----------|------|-----------|---------|------|
| 1 | ... | 无 | 5 | 30min | 待执行 |
| 2 | ... | 子任务1 | 8 | 40min | 待执行 |
| 3 | ... | 子任务2 | 3 | 20min | 待执行 |

### 执行建议
- 子任务 1-2 必须按顺序执行
- 子任务 3 可在子任务 2 完成后并行执行
```
