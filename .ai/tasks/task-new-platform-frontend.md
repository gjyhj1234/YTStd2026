# 任务：重构前端 AI 提示词体系（DevExtreme Vue Application Template + UI Templates + dxdocs + axios + 极细业务提示词标准）

## 任务背景

当前仓库中的前端提示词体系已经不能满足项目要求。现有问题包括但不限于：

1. 未严格限定前端模板来源，导致 AI 对“Application Template / UI Templates”理解模糊
2. DevExtreme Vue 与 dxdocs 约束不够具体，AI 仍会凭印象实现
3. HTTP 方案原先不再有参考意义，新前端默认统一采用 axios，但提示词中没有一套足够标准化、可执行、可复用的 axios 实现规范
4. 业务提示词拆分不够细，常出现：
   - 页面功能点不全
   - 表单字段遗漏
   - 查询条件遗漏
   - 操作按钮遗漏
   - 图标未统一
   - 权限控制遗漏
   - 分页缺失或不完整
   - 高级查询缺失
   - 静态配置未拆分
   - 国际化不完整
   - 验收只写“编译通过”，无法通过 code review 判断功能是否完整
5. 旧的前端任务进度说明已无价值，需要删除
6. 历史问题需要沉淀为新的“防错规则/反模式清单”
7. 需要支持 GitHub Agents 多轮、并行、可续接执行

本次任务的目标不是实现前端代码，而是**重构 `.ai/prompts/03-frontend/` 和 `.ai/prompts/08-platform/frontend/` 下的前端提示词体系**，让后续 GitHub Agents 能依据新的提示词体系重新构建前端。

---

## 总体目标

请重构现有前端提示词体系，形成一套新的、强约束、可执行、支持多轮续接和并行拆分的前端开发提示词标准。

   - 错误码翻译与国际化提示映射
   - 401 / 403 / 500 / 网络异常处理
   - 上传下载处理
   - 请求取消
   - 防重复提交
   - 并发请求约束（如需要）
   - 按钮 loading / 页面 loading / 表单 submitting 状态协作
   - 可 code-review 验证的检查点

- 是否支持回车搜索

- 查询条件文本是否全部国际化
- 查询相关静态配置是否拆分
- 查询条件是否纳入 code-review 验收

#### 3. 列表与分页必须明确
必须具体写明：
- 表格使用哪个 DevExtreme Vue 组件

- 是否支持空状态
- 是否支持 loading 状态
- 所有列标题
- 每列的数据来源
- 每列格式化规则
- 状态列是否颜色区分
- 操作列有哪些按钮
- 每个操作按钮图标是什么
- 每个操作按钮权限是否受控

必须具体写明每个按钮：
- 名称

- 图标
- 所属区域（工具栏 / 行操作 / 弹窗底部）
- 权限控制规则
- 启用/禁用条件

- 是否需要确认框
- 确认框文案是否国际化
- 成功/失败提示是否国际化

#### 5. 表单功能必须明确
每个表单必须逐项明确：
- 表单标题
- 使用的组件
- 字段顺序
- 每个字段名称
- 每个字段标签
- 每个字段类型
- 每个字段默认值
- 每个字段是否必填
- 每个字段长度限制
- 每个字段格式校验
- 每个字段唯一性校验
- 编辑场景下唯一性校验是否排除当前 Id
- 字段显示/隐藏条件
- 字段禁用条件
- 提示文本、placeholder、helpText 是否国际化
- 提交按钮、取消按钮、关闭按钮是否国际化
- 提交前校验
- 提交时 loading
- 提交成功后的行为
- 提交失败后的行为

#### 6. 验收标准必须是功能点 checklist
不能只写“实现用户新增”“实现查询功能”“编译通过”。

必须写成逐项 checklist，例如：
- 用户名字段存在
- 用户名标签存在
- 用户名必填
- 用户名长度下限
- 用户名长度上限
- 用户名新增重复校验

- 分页包含 pageIndex/pageSize/total
- 操作列包含编辑、禁用、重置密码
- 编辑按钮图标为 xxx
- 删除按钮图标为 xxx
- 每个按钮权限码存在
- 每个按钮显隐逻辑存在
- 等等

#### 7. 验收优先级必须调整
业务提示词中必须明确：
1. **先验收功能点是否完整**
2. **再验收规则是否完整**
3. **再验收 i18n 是否完整**
4. **最后才是编译是否成功**

## 七、字符乱码检查要求（必须新增）

新提示词体系必须新增“字符与编码质量检查”要求，防止出现中文乱码或异常字符。

必须明确要求：

1. 所有新提示词文件必须检查是否存在中文乱码
2. 所有新提示词输出中不得包含类似：
   - 包含 U+FFFD 替换字符的文本（如中文字被截断产生的碎片）
   - UTF-8 截断产生的不可识别字符
   - 编码错误产生的多字节乱码
3. 所有 Agent 生成的代码、JSON、Markdown、语言文件中也必须避免字符乱码
4. 必须在提示词中加入专门检查项：
   - Markdown 文件内容无乱码
   - JSON 语言文件无乱码
   - 中文标题、字段标签、提示文案无乱码
   - 代码注释与字符串无乱码
5. 验收输出必须显式说明是否进行了乱码检查，以及结果
6. 如果发现乱码，必须先修复再进入下一轮
